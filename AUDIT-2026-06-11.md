# Reactilities Audit — 2026-06-11

Full prod-readiness audit: 101 agents, code/tests/packaging/API/SSR/docs/ecosystem.

Severity counts: see summary in chat. `verified=True` = adversarially confirmed; `None` = medium/low (not re-verified); items in "Unverified high/critical" section had verifiers killed by session limit — finder evidence (often mutation-tested) stands.


## Area: code

**code:utils-1**: Audited all 10 hooks. Three hooks are clean: useDebounce, useDebounceCallback, and useFileReader (correct cleanup, ref-based latest-callback handling, abort on unmount/reset); useDragAndDrop is essentially clean apart from a low-severity drag-counter wedge. The serious problems cluster in the async layer: useFetch's documented refetch-on-url-change does not exist (useAsync only executes on mount), aborted previous requests are dispatched into the error state (visible on every mount under StrictMode), and the promised abort-on-unmount is unimplemented; useAsync also lacks any latest-request token, allowing out-of-order results. useCountdown's increment mode is broken with the default countStop (its own stopwatch example stops at 0 after one tick), useAutoSave's first-render guard fires a spurious save of unchanged data under StrictMode, and useEventSource/useGeolocation have state-staleness and ignored-option contract bugs.

**code:utils-2**: Audited all 10 hooks. useInterval, useIsMounted, and useManualUpdate are clean; useIdleTimeout and useImageLazyLoad are structurally sound with only low-severity behavioral gaps. The serious problems cluster in SSR safety (useKeyboardShortcuts evaluates `document` during render and will crash any Next.js server render; useNetworkState's getServerSnapshot throws, making it unusable even in server-rendered 'use client' components), an infinite observer-churn loop in useIntersectionObserver when an inline array threshold is passed, a double-fetch race in useInfiniteScroll whose loading guard lives in a stale closure, and a verified TS compile error: the RefObject&lt;HTMLElement | null&gt; returned by useIntersectionObserver/useInfiniteScroll is not assignable to any concrete element's ref prop, so the hooks' own JSDoc examples do not typecheck.

**code:utils-3**: Audited all 10 hooks. The most serious problems are in useWebSocket (zombie reconnect after unmount/URL change — also breaks StrictMode — and a reconnect-every-render loop when protocols is an inline array) and useThrottle (verified by simulation: with continuously changing values from mount, the throttled value never updates at all — it behaves like a debounce that never fires). useWorker and useVirtualization both have API contract bugs where their own JSDoc examples fail to compile under the library's strict tsconfig (verified with tsc), and usePermission leaks a change listener via an un-cancelled promise race. useTimeout is clean; useThrottleCallback and useWhyDidYouRender are essentially clean apart from one narrow edge case each; useSpeechRecognition and useSpeechSynthesis have only low-severity browser-edge issues.

**code:dom-1**: Audited all 11 DOM hooks plus their internal dependencies (useLocalStorage, useMediaQuery, useIsomorphicLayoutEffect), verifying type-level claims by compiling against the project's installed @types/react 19. The most serious problems: useLockBodyScroll can permanently lock page scroll with stacked overlays and clobbers stylesheet overflow rules; useClickOutside and useFocusTrap have ref typings whose own JSDoc examples fail to compile under React 19 types (verified TS2345/TS2322, masked in tests by seeding refs and an `as any` cast); and useDarkMode hard-crashes during SSR because its underlying stores throw in getServerSnapshot. Mid-tier issues include useLongPress timers surviving unmount and multi-touch, useFocusTrap being escapable once focus leaves the container, click-vs-mousedown misbehavior in useClickOutside, and stale-element bugs in useEventListener/useHover. useDocumentTitle and useFavicon are clean; useBattery's async/cleanup/StrictMode handling is correct apart from an SSR hydration mismatch on isSupported.

**code:dom-2**: Audited all 10 DOM hooks. Two high-impact correctness bugs: useOrientation's angle-based heuristic misreports landscape as portrait on every desktop monitor and on landscape-secondary (angle 270) devices, and useRovingTabIndex stores the focused index in a ref so the rendered tabIndex attributes never update — defeating the roving-tabindex pattern it exists to implement. useMediaQuery throws from getServerSnapshot, crashing any Next.js SSR pass (including 'use client' components) while its docstring claims SSR safety. Medium issues: useOrientation hydration mismatch/tearing (useState+effect instead of useSyncExternalStore) and useResizeObserver's static-ref effect that never observes conditionally/late-mounted elements. useMousePosition, usePageVisibility, useScrollPosition, and useWindowSize are clean; usePortal and useScrollTo have only low-severity footguns.

**code:state**: Audited all 15 state hooks. The serious problems cluster in the storage and cookie hooks: useLocalStorage/useSessionStorage share three real bugs (functional updaters losing rapid successive updates via a stale snapshot closure, a new parsed-object identity returned every render that can drive effect loops, and a throwing getServerSnapshot that crashes Next.js SSR), and useCookie ignores `name` changes, causes SSR hydration mismatches, and doesn't sync across instances despite its docs claiming synchronization. The clipboard pair has smaller contract bugs (useClipboard reports success when execCommand fails; useCopyToClipboard leaks its fallback textarea on throw and documents `await` on a void function). useBoolean, useToggle, useCounter, useList, useMap, useSet, useObjectState, and usePrevious are clean; useStep and useUndoRedo have only low-severity input-validation/unbounded-history notes.

**code:lifecycles-helpers**: Audited 8 modules. useIsomorphicLayoutEffect is clean (correct standard pattern), and both barrel files export correctly. The real problems: componentWillUnmount has a classic stale-closure bug (cleanup sees first-render state), useUpdateEffect's mount-skip is defeated by StrictMode's double-invoke in dev, and classnames diverges from clsx on falsy primitives (emits "0"/"NaN" classes and double spaces) despite being exported under the clsx/cn aliases. The remaining lifecycle shims and loadScript have API-contract issues (componentDidUpdate fires on mount, componentWillMount runs after render and warns in SSR, cleanup return values silently dropped, loadScript stringifies event-handler props).


### [CRITICAL ✓verified] useWebSocket: socket reconnects itself after unmount or URL change (zombie connections, StrictMode double-connect)
- **File:** src/utils/useWebSocket/index.ts:114-128, 160-167
- **Detail:** The effect cleanup calls `webSocketRef.current.close()` but never detaches handlers or signals an intentional close. `close()` fires the socket's `onclose` asynchronously AFTER cleanup has run; the handler then evaluates `shouldReconnectRef.current(event)` (default `() => true`) and `reconnectAttemptsRef.current < reconnectAttempts` (0 < 3), so it schedules `setTimeout(() => connectWebSocket(), reconnectInterval)`. Consequences: (1) on unmount, a brand-new WebSocket is created with no cleanup left to close it — if the server later closes it, it reconnects again, up to reconnectAttempts zombie connections; (2) on `socketUrl` change, the old socket's onclose closure captures the OLD `connectWebSocket`, so it reconnects to the stale URL in parallel with the new connection and overwrites `webSocketRef`, making `sendMessage` target the wrong socket; (3) in React 18 StrictMode dev, mount→cleanup→remount produces two live connections and doubled onMessage callbacks. The `clearTimeout(reconnectTimeoutRef.current)` in cleanup is useless here because the timeout is scheduled after cleanup runs.
- **Fix:** In the effect cleanup, mark the close as intentional before closing: e.g. set a `manuallyClosedRef.current = true` (or null out `socket.onclose/onmessage/onerror`) and check that flag in the onclose handler before scheduling a reconnect. Also reset `reconnectAttemptsRef` when socketUrl changes.


### [HIGH ✓verified] useFetch never re-fetches when url/options change, contradicting its own documented example
- **File:** src/utils/useFetch/index.ts:29-36, 54-79 (root cause: src/utils/useAsync/index.ts:124-127)
- **Detail:** The JSDoc example 'Re-fetch when params change' shows `useFetch(`/api/search?q=${query}`)` and claims it refetches as query changes. It cannot: useFetch delegates to useAsync, whose immediate-execute effect is `useEffect(() => { if (immediate) execute(); }, [])` with an empty dep array, and `execute` reads the function through `asyncFnRef` without ever re-running. A new `url` updates `fetchFn` and the ref, but nothing triggers a fetch. A SearchResults component built per the example silently shows the first query's results forever. The useFetch test file never tests a rerender with a changed url, so this is unguarded.
- **Fix:** In useAsync, re-run the immediate effect when the async function identity changes (e.g. deps `[immediate, asyncFn]` with abort/ignore of the stale call), or have useFetch own an effect keyed on `[url, options]` that calls execute. Also note `options` is an object — compare via serialized key or instruct consumers to memoize, otherwise refetch-on-change would loop with inline option objects.


### [HIGH ✓verified] useFetch surfaces AbortError of the cancelled previous request as the error state; in StrictMode every mount shows 'error' while the real request is in flight
- **File:** src/utils/useFetch/index.ts:62-65 with src/utils/useAsync/index.ts:115-118
- **Detail:** fetchFn aborts the previous controller before starting a new request. The aborted fetch rejects with a DOMException (instanceof Error), and useAsync's catch dispatches `{ type: 'error' }` whenever still mounted — there is no check that this result belongs to the latest execute(). Sequence: execute#2 dispatches 'loading', aborts fetch A; A rejects -> state becomes 'error' (AbortError) and STAYS 'error' until fetch B settles. Any consumer rendering `if (error) return <ErrorMessage/>` shows an abort error for the entire duration of the new request. In React StrictMode (default in Next.js/CRA dev) this happens automatically on every mount because the immediate effect double-fires execute().
- **Fix:** Ignore rejections whose name is 'AbortError' (or whose signal is aborted), and/or add a request-id/token in useAsync so only the latest execute() may dispatch success/error.


### [HIGH ✓verified] useCountdown increment mode is broken with the default countStop — the hook's own stopwatch example never counts
- **File:** src/utils/useCountdown/index.ts:91, 109-121 (example at 72-86)
- **Detail:** `countStop` defaults to 0. In the interval tick, `isDone = isIncrement ? next >= countStop : next <= countStop`. For the documented stopwatch example `useCountdown({ countStart: 0, isIncrement: true })`: first tick computes next=1, isDone = 1 >= 0 = true, so it immediately clamps count back to 0 and stops. The stopwatch displays 0 forever and isRunning flips false after one interval. Any increment usage where countStop <= countStart (which the default guarantees for non-negative starts) terminates instantly. The unit test masks this by always passing `countStop: 5` explicitly.
- **Fix:** Default countStop to `Infinity` when `isIncrement` is true (or validate/document that countStop must exceed countStart in increment mode), and fix the JSDoc stopwatch example. Add a test for increment with default countStop.


### [HIGH ✓verified] useKeyboardShortcuts crashes during SSR: `document` evaluated as default value at render time
- **File:** src/utils/useKeyboardShortcuts/index.ts:54
- **Detail:** `const { target = document, enabled = true } = options;` runs in the hook body, i.e. during render. In Node/Next.js SSR `document` is not defined, so any component calling `useKeyboardShortcuts(...)` without an explicit `target` throws `ReferenceError: document is not defined` during server render. This includes 'use client' components, which Next.js still server-renders. The consumer cannot even work around it by passing `target: document`, since referencing `document` in their render has the same problem.
- **Fix:** Default `target` to `undefined` in the destructuring and resolve it inside the effect: `const eventTarget = target ?? document;` within `useEffect`, where `document` is guaranteed to exist. Keep `target` in the dep array.


### [HIGH ✓verified] useThrottle: value never updates while it keeps changing (behaves as a never-firing debounce from mount)
- **File:** src/utils/useThrottle/index.ts:37-47
- **Detail:** `if (lastUpdated.current && now >= lastUpdated.current + interval)` — `lastUpdated.current` starts as `null` and is only ever set inside the timeout callback or the immediate branch. On every change the else branch schedules `setTimeout(..., interval)` (full interval, not remaining time), and the next change's cleanup clears it. So if values change more often than `interval` starting from mount (the hook's headline use case: scroll/mouse tracking), no timer ever fires, `lastUpdated` stays null, and the immediate branch is permanently unreachable. Verified by simulation: with value changing every 100 ms and interval 500 ms from t=0, throttledValue is still the initial value after 3 s of changes — zero updates until the changes pause for a full interval. A throttle must emit roughly once per interval during sustained changes.
- **Fix:** Change the condition to `if (lastUpdated.current === null || now >= lastUpdated.current + interval)` so the first change passes through (leading edge), and schedule the trailing timeout for the remaining time (`lastUpdated.current + interval - now`) instead of the full interval.


### [HIGH ✓verified] useWebSocket: inline `protocols` array causes disconnect/reconnect on every render
- **File:** src/utils/useWebSocket/index.ts:153, 155-168
- **Detail:** `connectWebSocket` is memoized with `[socketUrl, protocols, reconnectAttempts, reconnectInterval]` and the connect effect depends on `connectWebSocket`. `protocols?: string | string[]` — a consumer passing `useWebSocket(url, { protocols: ['graphql-ws'] })` creates a new array reference every render, so `connectWebSocket` is recreated and the effect tears down and reopens the socket on EVERY render of the component. The hook's own comment (lines 71-73) shows the authors fixed exactly this for the callbacks but missed `protocols`. Every render of a chatty component (e.g. one that re-renders on each incoming message via `lastMessage`) becomes a connect/close loop.
- **Fix:** Stabilize protocols: e.g. depend on `JSON.stringify(protocols)` / `Array.isArray(protocols) ? protocols.join(',') : protocols`, or hold it in a ref like the callbacks (documenting that protocol changes don't reconnect).


### [HIGH ✓verified] useLockBodyScroll: restores computed style as inline style; nested locks permanently lock the page
- **File:** src/dom/useLockBodyScroll/index.ts:33-37
- **Detail:** The effect saves `window.getComputedStyle(document.body).overflow` and on cleanup writes that COMPUTED value back as an INLINE style. Two failures: (1) Nested locks (the hook's primary use case: stacked modal + drawer): A mounts (saves 'visible', sets hidden), B mounts (saves computed 'hidden'!, sets hidden). Close A first -> body restored to 'visible' while B is still open (scroll unlocked behind the modal). Then close B -> restores 'hidden' -> body is PERMANENTLY overflow:hidden and the page can never scroll again. (2) Even single use writes e.g. inline `overflow: visible` on cleanup where there was previously no inline style, permanently overriding any stylesheet/media-query rule on body.
- **Fix:** Save and restore the inline value `document.body.style.overflow` (which is '' when unset), not the computed style. Better: keep a module-level reference count so the style is only restored when the last lock releases.


### [HIGH ✓verified] useOrientation reports 'portrait' on desktops and on landscape-secondary devices (angle-based heuristic is wrong)
- **File:** src/dom/useOrientation/index.ts:15-16
- **Detail:** Orientation is derived as `Math.abs(Number(angle)) === 90 ? 'landscape' : 'portrait'`. Two concrete failure modes: (1) `screen.orientation.angle` is 0/90/180/270 — a phone rotated to landscape-secondary reports angle 270, so `Math.abs(270) === 90` is false and the hook returns 'portrait' while the device is in landscape. (2) On any landscape-natural device (every desktop monitor, many tablets) `screen.orientation.angle` is 0 in landscape, so the hook reports 'portrait' on a 1920x1080 desktop screen. The angle only encodes rotation from the device's natural orientation, not portrait vs landscape. The legacy `window.orientation` values (0/±90/180) are the only case the heuristic handles correctly.
- **Fix:** Derive the type from `screen.orientation.type.startsWith('landscape')` when available, or from `window.matchMedia('(orientation: landscape)').matches`, falling back to the angle heuristic only for the legacy `window.orientation` path. Keep `angle` as-is in the return value.


### [HIGH ✓verified] useRovingTabIndex: tabIndex never actually roves — focused index stored in a ref, so no re-render updates the DOM
- **File:** src/dom/useRovingTabIndex/index.ts:54,104-105
- **Detail:** `focusedIndexRef` is a useRef (line 54) and `getItemProps` computes `tabIndex: index === focusedIndexRef.current ? 0 : -1` (line 104) at render time. Arrow-key navigation (`focusItem`) and `onFocus` only mutate the ref — nothing triggers a re-render, so the rendered `tabIndex` attributes stay frozen at whatever the last render produced (item 0 keeps tabIndex=0, the newly focused item keeps -1). This defeats the entire roving-tabindex pattern: after the user arrows to item 3, Tabs out of the widget, and Shift+Tabs back, focus lands on item 0 instead of item 3. The WAI-ARIA contract promised in the docstring ("Only the focused item is in the tab order") is only true on the initial render.
- **Fix:** Store focusedIndex in useState (or useReducer) so a focus change re-renders consumers and updates the DOM tabIndex attributes; keep a ref mirror if needed inside handleKeyDown to avoid stale closures, or just include the state in deps.


### [HIGH ✓verified] useLocalStorage/useSessionStorage: returned value is a new object reference every render (JSON.parse not memoized)
- **File:** src/state/useLocalStorage/index.ts:116-125 (identical bug in src/state/useSessionStorage/index.ts:119-128)
- **Detail:** `const parsedValue = (() => { ... return JSON.parse(store); })();` runs on every render, so for any non-primitive T the hook returns a brand-new object/array identity each render even when nothing changed. A consumer doing `const [user] = useLocalStorage('user', {...}); useEffect(() => {...}, [user])` gets the effect firing on every render; if that effect setStates, it's an infinite re-render loop. This defeats the stability that useSyncExternalStore's string snapshot provides.
- **Fix:** Memoize the parse on the snapshot string: `const parsedValue = useMemo(() => { if (store === null) return initialValueRef.current; try { return JSON.parse(store); } catch { return initialValueRef.current; } }, [store]);`.


### [HIGH ✓verified] componentWillUnmount cleanup runs with stale first-render closure
- **File:** src/lifecycles/componentWillUnmount/index.ts:40-43
- **Detail:** `useEffect(() => { return () => { func(); }; }, [])` stores only the `func` passed on the first render. The hook is called every render with a fresh closure, but the empty dep array means the closure captured at mount is the one invoked at unmount. Consumer code like `componentWillUnmount(() => saveDraft(text))` will always save the initial `text` (e.g. '') instead of the latest value — exactly the use case the JSDoc examples advertise ('Save data before unmount'). The bug is silent: no error, just wrong data.
- **Fix:** Keep the latest callback in a ref updated on every render and invoke the ref in the cleanup: `const fnRef = useRef(func); fnRef.current = func; useEffect(() => () => { fnRef.current(); }, []);` (this is the pattern react-use's useUnmount uses).


### [MEDIUM] useFetch does not abort the in-flight request on unmount despite documenting 'automatic abort on re-fetch or unmount'
- **File:** src/utils/useFetch/index.ts:6, 59-79
- **Detail:** The hook contains no effect at all; `abortControllerRef.current.abort()` is only called at the start of the next fetchFn invocation. On unmount nothing aborts the controller, so the network request runs to completion (wasted bandwidth, server load; only the setState is suppressed by useAsync's mountedRef). The JSDoc at line 6 explicitly promises abort on unmount.
- **Fix:** Add `useEffect(() => () => abortControllerRef.current?.abort(), [])` in useFetch.


### [MEDIUM] useAutoSave fires a spurious save of unchanged initial data under React StrictMode
- **File:** src/utils/useAutoSave/index.ts:61, 78-92
- **Detail:** The 'skip first render' guard uses `isFirstRender` ref set to false inside the effect, with no reset in cleanup (the first-render branch returns no cleanup at all). In StrictMode dev (mount -> cleanup -> remount), the second effect invocation sees `isFirstRender.current === false`, sets status to 'pending', and schedules `setTimeout(save, delay)` — so `onSave(initialData)` is called ~delay ms after mount even though the data never changed. That's a real server write triggered in every dev session of a Next.js/CRA app, and it breaks under any future remount semantics (Activity/offscreen). Additionally, because the effect deps are `[data, delay, save]`, changing only `delay` also triggers a 'pending' + save of unchanged data.
- **Fix:** Track the last-seen data instead of a first-render flag: e.g. keep `prevDataRef` and only schedule a save when `data !== prevDataRef.current` (updating the ref in the effect). This is StrictMode-safe and also stops delay changes from causing spurious saves.


### [MEDIUM] useAsync has no protection against out-of-order results from overlapping execute() calls
- **File:** src/utils/useAsync/index.ts:110-120
- **Detail:** `execute` only guards with `mountedRef`. If execute() is called while a previous call is pending (refresh button, StrictMode double-mount with `immediate`), both promises dispatch on settle; whichever settles LAST wins, so an older, slower response can overwrite a newer result with stale data (and an old failure can overwrite a new success with 'error'). The JSDoc Refresh-button example is exactly this scenario.
- **Fix:** Keep a monotonically increasing call id (ref); capture it at the start of execute and only dispatch if it still equals the latest id when the promise settles.


### [MEDIUM] useEventSource: setting url to null leaves readyState/lastMessage stale — UI keeps showing 'Connected' and old data after the connection is gone
- **File:** src/utils/useEventSource/index.ts:56-57, 75-76, 96-98
- **Detail:** The effect starts with `if (!url) return;`. When url transitions from a string to null, the previous effect's cleanup closes the EventSource, but the new effect run does nothing — `readyState` stays at its last value (e.g. 1 OPEN), so the documented `readyState === 1 ? 'Connected'` check renders 'Connected' with no live connection. `lastMessage` also persists across url changes (including to a different endpoint), so consumers can render data from the previous stream. Related: with url=null from the start, readyState is 0 (CONNECTING) forever instead of 2 (CLOSED).
- **Fix:** In the `!url` branch set `setReadyState(2)` (and consider initializing readyState based on url being null). Clear or reset `lastMessage` when the effect re-runs for a new url, matching the documented close() behavior.


### [MEDIUM] useGeolocation: options changes are silently ignored, and the code comment claiming otherwise is wrong
- **File:** src/utils/useGeolocation/index.ts:72-75, 106-121
- **Detail:** The comment says optionsRef is kept current 'so changes to options take effect on the next watchPosition call', but watchPosition (and getCurrentPosition) are invoked exactly once in a `[]`-deps effect — there is no next call. Changing `enableHighAccuracy`, `timeout`, or `maximumAge` after mount has no effect whatsoever; the watch keeps the original options forever. A consumer toggling high-accuracy mode gets nothing, with no error.
- **Fix:** Re-register the watch when options change: depend on the primitive option values (`[options.enableHighAccuracy, options.timeout, options.maximumAge]`) and clearWatch/re-watch in the effect. Otherwise document that options are mount-only and delete the misleading comment.


### [MEDIUM] useNetworkState throws on the server and during hydration, breaking all SSR apps
- **File:** src/utils/useNetworkState/index.ts:83-85,147
- **Detail:** `getServerSnapshot` is `() => { throw Error("useNetworkState is a client-only hook"); }`. React calls getServerSnapshot both on the server render AND on the client during the hydration render. So in any Next.js/Remix app the hook throws even inside a 'use client' component; the only escape is `dynamic(..., { ssr: false })` or a mounted-flag wrapper. For a public library this makes the hook effectively unusable in the most common React deployment target. The rest of the hook (module-level stable `subscribe`, ref-cached snapshot with shallow-equal to keep referential stability) is correct.
- **Fix:** Return a stable module-level default snapshot from getServerSnapshot, e.g. `const SERVER_SNAPSHOT: NetworkState = { online: true }; const getServerSnapshot = () => SERVER_SNAPSHOT;`. The client snapshot replaces it right after hydration. Document that the server assumes online.


### [MEDIUM] useIntersectionObserver: array `threshold` in dep array causes effect to re-run every render and an infinite re-render loop
- **File:** src/utils/useIntersectionObserver/index.ts:61
- **Detail:** Effect deps are `[frozen, observerOptions.threshold, observerOptions.root, observerOptions.rootMargin]`. `threshold` is typed `number | number[]` (from IntersectionObserverInit). With the very common inline call `useIntersectionObserver({ threshold: [0, 0.5, 1] })`, the array has a new identity every render, so the effect tears down and recreates the observer on every render. Worse: every newly created IntersectionObserver fires its callback once with a fresh entry object, so `setEntry(entry)` always sets a new object -> re-render -> new array identity -> new observer -> callback -> setEntry... an endless async render/observe loop plus continuous observer churn.
- **Fix:** Normalize the threshold in the dep array, e.g. `const thresholdKey = Array.isArray(observerOptions.threshold) ? observerOptions.threshold.join(',') : observerOptions.threshold;` and depend on `thresholdKey` while constructing the observer with the original value (kept in a ref or re-read from options).


### [MEDIUM] Returned ref type RefObject<HTMLElement | null> does not typecheck against any concrete element (the hooks' own examples fail to compile)
- **File:** src/utils/useIntersectionObserver/index.ts:8,37 and src/utils/useInfiniteScroll/index.ts:18,59
- **Detail:** Both hooks return `RefObject<HTMLElement | null>`. Verified with the repo's @types/react 19.2.14 under strict mode: `<div ref={loaderRef} />` fails with `TS2322: Type 'RefObject<HTMLElement | null>' is not assignable to type 'Ref<HTMLDivElement> | undefined' ... Type 'HTMLElement | null' is not assignable to type 'HTMLDivElement | null'`. So the usage shown in both hooks' JSDoc (`<div ref={ref}>`, `<div ref={loaderRef}>`) is a compile error for every TypeScript consumer; they must cast. useImageLazyLoad avoids this by using HTMLImageElement.
- **Fix:** Make the hooks generic: `function useIntersectionObserver<T extends HTMLElement = HTMLElement>(...)` with `const ref = useRef<T>(null)` returning `RefObject<T | null>`, so consumers write `useIntersectionObserver<HTMLDivElement>()`. Same for useInfiniteScroll's loaderRef.


### [MEDIUM] useInfiniteScroll: loading guard is stale-closure React state, allowing duplicate onLoadMore calls
- **File:** src/utils/useInfiniteScroll/index.ts:69-85
- **Detail:** `handleIntersect` guards with `if (!entry.isIntersecting || !hasMore || isLoading) return;` where `isLoading` is React state captured in the useCallback closure (deps `[hasMore, isLoading]`). The guard only becomes effective after `setInternalLoading(true)` commits AND the effect re-runs to swap the observer. If the IntersectionObserver fires again before that commit (scroll jitter making the sentinel exit/re-enter, or a slow render blocking the commit), the same stale closure sees `isLoading === false` and invokes `onLoadMoreRef.current()` a second time — a duplicate page fetch, the classic infinite-scroll double-load bug. Note also that `setInternalLoading(true)` runs only after `onLoadMore()` has already been called and returned a promise, widening the window.
- **Fix:** Add a synchronous ref guard: `const inFlightRef = useRef(false);` check-and-set it at the top of handleIntersect before calling onLoadMore, clear it in `finally`. Keep the state for the returned `isLoading` flag.


### [MEDIUM] usePermission: promise race leaks a permanent change listener and applies stale state after cleanup
- **File:** src/utils/usePermission/index.ts:37-60
- **Detail:** The effect has no cancellation flag. If cleanup runs while `navigator.permissions.query()` is still pending (`permissionStatus` is still null, so `removeEventListener` is a no-op), the `.then` later runs anyway: it calls `setState(status.state)`/`setLoading(false)` and attaches `status.addEventListener('change', handleChange)` — a listener that is never removed (PermissionStatus objects are browser-global and long-lived). Concrete failures: (1) when `permissionName` changes mid-flight, the old permission's resolution overwrites the new permission's state, and every future change of the OLD permission keeps clobbering `state`; (2) in StrictMode dev the first effect's promise always resolves after its cleanup, leaking a duplicate listener on every mount; (3) the listener closure keeps setState alive after unmount indefinitely.
- **Fix:** Add a `let cancelled = false` flag set in cleanup; in `.then`, if cancelled, call `status.removeEventListener` immediately (or don't attach) and skip the setStates.


### [MEDIUM] useWorker: generic signature rejects any typed function — the documented example does not compile
- **File:** src/utils/useWorker/index.ts:9, 71
- **Detail:** `useWorker<T>(fn: (...args: unknown[]) => T)` — under `strict` (which this repo's tsconfig.lib.json enables), a typed function is not assignable to `(...args: unknown[]) => T`. Verified with the project's own tsc: the JSDoc example `useWorker((numbers: number[]) => numbers.reduce(...))` fails with TS2345 "Type 'unknown' is not assignable to type 'number[]'". Consumers must erase their parameter types to call the hook at all, and `run: (...args: unknown[]) => void` provides no type checking that the arguments match `fn`'s parameters.

## Area: ssr

**ssr-sweep**: SSR readiness is mixed: the library is import-safe in Node (verified — no module-scope browser access executes; all browser hits live in lazily-called subscribe/getSnapshot helpers or effects/handlers) and uses useSyncExternalStore well in 7 hooks, three of which (useWindowSize, useScrollPosition, usePageVisibility) ship proper server snapshots. ## Ecosystem research

Ecosystem snapshot (June 2026, downloads verified via api.npmjs.org last-week): The hooks-library market is large but its two biggest incumbents are coasting or dead. usehooks-ts (~4.4M/wk) is marked "inactive" on Snyk with no release in 12+ months, an open React 19 peer-dep issue (juliencrn/usehooks-ts#650, Nov 2024) and an open useLocalStorage hydration-mismatch issue (#644). react-use (~3.6M/wk) is effectively abandoned — the last active collaborator quit in 2021 (streamich/react-use#1974), with acknowledged SSR design flaws; downloads are pure legacy inertia. @uidotdev/usehooks (~970k/wk) won its HN launch on the "server-component-safe" angle (news.ycombinator.com/item?id=36129622) and has a beloved docs site (usehooks.com) with live demos, but the npm package hasn't shipped since 2023. The maintained players: @mantine/hooks (~2.1M/wk, v9.3.0 days ago, standalone, excellent docs, SSR-aware APIs like getInitialValueInEffect) and ahooks (~700k/wk, Alibaba, healthy cadence, enterprise/China-centric, had its own React 19 peer-dep friction, alibaba/hooks#2670). Notable newer entrants: @reactuses/core "ReactUse" (~190k/wk, 110+ hooks, VueUse-inspired, React 16.8–19, aggressive SEO comparison-blog strategy on reactuse.com) and react-haiku (~3.3k/wk, pretty docs site, small). reactilities sits at 12 downloads/wk.

What actually drives adoption in 2026 (from issue threads, HN/Reddit sentiment, and what worked for entrants): (1) a docs site with live demos — every library with traction has one; repo READMEs are invisible to search and to skimmers; (2) SSR/Next.js App Router correctness — hydration mismatch in useLocalStorage/useMediaQuery is the single most common complaint across usehooks-ts, react-use and ahooks, and "server-safe" was literally the pitch that got @uidotdev/usehooks to the HN front page; (3) trust signals an unknown author must externalize — CI badge, coverage upload, release cadence, fast issue responses, React 18/19 test matrix; (4) bundle proof — tree-shaking or per-hook entry points with visible sizes; (5) distribution flexibility — the shadcn registry now has a first-class `registry:hook` item type and 2025's Universal Registry Items extended it beyond components, so the copy-paste-ownership crowd (the main objection to hooks libraries: "I'll just copy 10 lines") can be served via `npx shadcn add` from your own registry while npm serves the rest; (6) launch channels: This Week in React (submit by replying to the newsletter), r/reactjs Show-off threads, Show HN with a sharp angle, ui.dev Bytes.

Realistic wedge for reactilities: it cannot win on hook count (74 vs react-use's 100+ and ReactUse's 110+) or on brand. The open seat is "the maintained, TypeScript-first, App-Router-safe standalone hooks library" — usehooks-ts's seat, which it is visibly vacating. reactilities already has real substance for this story (628 tests, ~99% coverage, useSyncExternalStore-based implementations, zero deps, sideEffects:false), but three gaps currently disqualify it before any launch: no docs site, no CI (the coverage badge is self-asserted), and no 'use client' banner in the dist output (verified: 0 occurrences), which means Next.js App Router users get confusing server-component errors — the exact failure mode the market is angriest about. Fix the must-haves below, then launch on the "maintained + server-safe alternative" angle; do not launch before, because for an unknown author the first impression on HN/r/reactjs is the only one. Sources: https://snyk.io/advisor/npm-package/usehooks-ts, https://github.com/juliencrn/usehooks-ts/issues/650, https://github.com/juliencrn/usehooks-ts/issues/644, https://github.com/streamich/react-use/issues/1974, https://github.com/uidotdev/usehooks, https://news.ycombinator.com/item?id=36129622, https://mantine.dev/hooks/package/, https://github.com/alibaba/hooks/issues/2670, https://reactuse.com/, https://ui.shadcn.com/docs/registry, https://thisweekinreact.com/newsletter.


### [MUST] Ship a docs site with a live demo per hook
Non-negotiable: usehooks.com, usehooks-ts.com, mantine.dev, reactuse.com and even tiny react-haiku all have one; reactilities' per-hook READMEs inside src/ are invisible to Google and to evaluating developers. Use Nextra/Starlight/Docusaurus on Vercel or GitHub Pages with: searchable hook index grouped by category (state/dom/utils/lifecycle), an interactive demo + copyable snippet per hook, and an explicit per-hook SSR-behavior note (the thing nobody else documents systematically). This is also a hard prerequisite for any launch post.


### [MUST] Add 'use client' to the build output and make App Router safety the headline claim
Verified: dist/reactilities.js contains zero 'use client' directives, so importing any hook from a Next.js App Router server component fails with a confusing error. Add the directive via Vite banner config, then audit and document every hook for hydration safety (useLocalStorage/useMediaQuery/useWindowSize style hooks must render a consistent server snapshot — reactilities already uses useSyncExternalStore, which is the right primitive; say so loudly). Hydration mismatch is the most-complained-about bug across usehooks-ts (#644), react-use, and ahooks (#2856), and 'server-safe hooks' is the exact pitch that put @uidotdev/usehooks on the HN front page in 2023. This is reactilities' single strongest technical differentiator against the inactive incumbents.


### [MUST] Add CI and externalize trust signals (the unknown-author tax)
There is no .github directory; the README's 99%-coverage badge is a static self-asserted shield, which sophisticated evaluators discount. Add GitHub Actions: lint + typecheck + the 628-test suite on a React 18 and React 19 matrix, coverage uploaded to Codecov (dynamic badge), npm publish via changesets with provenance (--provenance gives the npm 'built and signed on GitHub Actions' checkmark — strong signal for an unknown publisher). Add a CHANGELOG and commit to a visible release cadence. For a single-author library in 2026, every trust claim must be machine-verified, not asserted.


### [MUST] Explicit React 19 support, tested and stated
peerDependencies react>=18 technically allows 19, but incumbents' open React 19 issues (usehooks-ts #650 unresolved since Nov 2024; ahooks #2670) are an active migration pain point — teams upgrading to React 19 are precisely the ones shopping for a replacement library. Run the test matrix against React 19, state 'React 18 & 19, no peer-dep warnings' in the README and docs hero, and avoid APIs deprecated in 19.


### [SHOULD] Per-hook subpath exports plus public bundle-size proof
Current package.json exposes a single '.' entry with an ESM bundle and a UMD/CJS file; sideEffects:false makes ESM tree-shaking work in modern bundlers, but it's unprovable to a skeptical evaluator and the CJS path doesn't tree-shake at all. Add per-hook subpath exports ('reactilities/useClickOutside'), drop or de-emphasize UMD, and publish per-hook sizes (bundlejs.com links or a size-limit CI check with badge). 'Import one hook, pay for one hook' with receipts directly counters the copy-paste objection.


### [SHOULD] Publish a shadcn-compatible registry for copy-paste installation
The shadcn registry has a first-class registry:hook item type, and 2025's Universal Registry Items made registries a mainstream distribution channel beyond UI components. Hosting a registry.json so users can `npx shadcn add https://reactilities.dev/r/use-click-outside` converts the largest competitor — developers who refuse hooks dependencies on principle and copy-paste instead — into users. Offering both npm and source-ownership installation is something none of the big six hooks libraries currently does, making it a genuinely novel positioning point for the launch post.


### [SHOULD] Position explicitly against the dying incumbents with a comparison/migration page
The realistic wedge: usehooks-ts (~4.4M/wk) is inactive 12+ months, react-use (~3.6M/wk) is abandoned since ~2021, @uidotdev/usehooks hasn't published since 2023 — millions of weekly downloads are flowing to unmaintained code. Build an honest comparison table (maintenance status, React 19, SSR/'use client', tree-shaking, per-hook docs) plus a hook-name migration map from react-use/usehooks-ts to reactilities. @reactuses/core is proving this works with its SEO comparison-blog strategy ('Best React Hooks Libraries in 2026' ranks for the money queries). Frame as 'a maintained, App-Router-safe alternative' — never trash-talk; the data speaks.


### [SHOULD] Sequenced launch plan across React-specific channels
Only after the musts ship, ideally tagged v1.0: (1) This Week in React — Sébastien Lorber accepts submissions via newsletter reply; a featured 'new library' slot reaches the most experienced React audience; (2) r/reactjs weekend show-off thread with the maintenance-gap framing; (3) Show HN titled around the differentiator ('Show HN: Maintained, server-safe React hooks — an alternative to react-use/usehooks-ts'), since 'server-safe' already front-paged once; (4) ui.dev Bytes and React Status submissions; (5) an X/Bluesky thread with a 30-second demo GIF. Stagger over 2–3 weeks and be hyper-responsive to every issue filed during the window — first-week issue-response time is itself a marketed trust signal.


### [NICE] JSDoc on every hook plus llms.txt for AI-assistant discovery
In 2026 a large share of library discovery happens inside AI coding assistants, which favor libraries with rich inline JSDoc (surfaces in IntelliSense and in model context) and crawlable docs. Add a JSDoc block with description, @param/@returns and an @example to every exported hook, and publish llms.txt / llms-full.txt on the docs site. Incumbents win by being in training data; being maximally legible to retrieval is the counter available to a new entrant.

None of this drives adoption alone, but it removes the 'side project' smell that makes evaluators bounce from unknown-author libraries.
